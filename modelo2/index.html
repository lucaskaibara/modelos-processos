Modelos de Processos TRABALHO DE ENGENHARIA DE SOFTWARE

INTRODUÇÃO AO CONTEÚDO

Os modelos de processos fornecem um guia específico para o trabalho da engenharia de software. Este guia define o fluxo de todas as atividades, ações e tarefas, o grau de iteração, os artefatos e a organização do trabalho a ser feito.

Estes processos trazem estabilidade, controle e organização para uma atividade que pode, sem controle, tornar-se bastante caótica. Entretanto, uma abordagem de engenharia de software moderna deve ser ágil, devendo sempre demandar apenas atividades, controles e produtos de trabalho que sejam apropriados para a equipe do projeto e para o produto a ser gerado.

Tipos de Modelos de Processos

CASCATA - INCREMENTAL - EVOLUCIONÁRIO - CONCORRENTE

Cascata

É o modelo clássico, sugere uma abordagem sequencial e sistemática para o desenvolvimento. É melhor utilizado quando os requisitos são bem definidos e estáveis. Problemas do modelo: Projetos reais raramente seguem um fluxo sequencial, é difícil para o cliente estabelecer explicitamente todas as necessidades, O cliente deve ter paciência, pois uma versão funcional só estará disponível próximo ao final do projeto.

Incremental

Seu cliente exige a entrega em uma data impossível de atender. Sugira entregar um ou mais incrementos nessa data e o restante do software (incrementos adicionais) posteriormente. O modelo incremental libera uma série de versões, denominadas incrementos, que oferecem, progressivamente, maior funcionalidade ao cliente à medida que cada incremento é entregue Modelo Incremental.

O primeiro incremento é um produto essencial, ou seja, os requisitos básicos são atendidos; porém, muitos recursos complementares (alguns conhecidos, outros não) ainda não são entregues. Esse processo é repetido após a liberação de cada incremento até que seja gerado o produto completo.

Evolucionário

Os modelos de processo evolucionário produzem uma versão cada vez mais completa do software a cada iteração.

Prototipação

Quando seu cliente tiver uma necessidade legítima, mas sem a mínima ideia em relação aos detalhes, faça um protótipo como uma primeira etapa. Problemas: os envolvidos encaram o protótipo como uma versão operacional, não considerando a qualidade e manutenção a longo prazo.

Espiral

Une a natureza iterativa da prototipação aos aspectos sistemáticos e controlados do modelo cascata. Tem potencial para o rápido desenvolvimento de versões cada vez mais completas do software. Pode ser adaptado para ser aplicado ao longo de todo o ciclo de vida de uma aplicação, desde o desenvolvimento de conceitos até sua manutenção.

Concorrente

Os modelos de processos concorrentes são bem utilizados quando se tem varias equipes diferentes de engenharia, eles são utilizados quando se necessita de uma visão clara do estado do projeto, nesse modelo todas as etapas do projeto existem simultaneamente porem em estados diferentes.
